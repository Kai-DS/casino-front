## 1. 目的（何を変えるのか）

### 1-1. 変更前（現行）

- レバーON時に内部抽選で成立役を決定する。
- 停止操作は形だけで、停止ボタンを押すと**押した位置に関係なく**、成立役に対応した出目へ自動的に揃う。
- 目押しの概念が存在しない。

### 1-2. 変更後（本書）

- レバーON時に内部抽選で成立役を決定する。**ここは変更しない。**
- 停止ボタンを押した瞬間の「各リールの現在位置」を基準に、**最大すべりコマ数（既定4コマ）以内**でリールを滑らせて停止させる。
- 成立している役は、すべり範囲内に揃えられる位置があれば**必ず引き込んで揃える**（引き込み制御）。
- 成立していない役・ボーナスは、すべり範囲をどう使っても**絶対に揃えない**（蹴飛ばし制御）。
- 結果として、プレイヤーの停止位置（＝目押し）が停止出目に影響する。ただし成立役の取りこぼし／不成立役の誤入賞は発生しない（実機の「100%引き込み・100%蹴飛ばし」を採用。§2-4 参照）。

### 1-3. 変えないもの（スコープ外）

- 内部抽選ロジック（lottery 層）・確率・設定差。
- 配当枚数・ボーナス内容・天井・RUSH/COUNTDOWN の進行ロジック。
- BET 仕様、液晶表示の内容。
- 演出区間（NEON COUNTDOWN、RUSH継続抽選フェーズ）の「BET不要・演出操作」という性質。これらは通常のリール回転とは別扱いのまま（§7参照）。

## 2. 用語と前提

### 2-1. コマ（symbol position）

リール1本は **21コマ**（既存仕様どおり）。各コマに図柄が1つ割り当てられる。コマ番号は **0-indexed（0〜20）** とする。

> 注: v7.2 のリール配列表は 1〜21 の 1-indexed・上から降順で記載されている。本書では実装上の都合で 0-indexed の配列インデックスを用いる。配列の「並び順」をどう持つかは §3-2 で確定する。**並び順の取り違えは出目バグに直結するため、§3-2 のテストを必ず通すこと。**

### 2-2. 表示窓（visible window）

画面に見えるのは各リール **3コマ**（上段・中段・下段）。本書では各リールの「現在位置」を **中段に表示されているコマのインデックス**で表す（§4-1）。

- 上段 = (中段 - 1 + 21) % 21
- 中段 = position
- 下段 = (中段 + 1) % 21

### 2-3. すべりコマ数（slip / pull-in count）

停止ボタンを押した瞬間の位置から、停止確定までにリールがさらに進むコマ数。

- **最大すべりコマ数 `MAX_SLIP = 4`**（既定。実機ジャグラー系準拠。設定で変更可能にするが既定は4）。
- すべりは **0〜MAX_SLIP コマ**の範囲。すべり0＝押した位置（厳密には押した瞬間の中段コマ）でそのまま停止。
- リールは一方向（コマ番号が増える向き）に回転する前提。すべりは常に「進む方向」へのみ発生する（戻り・逆回転すべりはしない）。

### 2-4. 引き込み・蹴飛ばしの強さ（本機の方針）

本機は **100%引き込み・100%蹴飛ばし** を採用する。

- 成立役は、すべり0〜4の5通りのいずれかで必ず揃えられるように図柄配置されていることを前提とする（§3-3 の配置制約・検証で保証）。
- もし配置の都合で「4コマ以内に引き込めない」ケースが存在する場合、それは**配列設計のバグ**として扱い、§3-3 の検証スクリプトで検出する。実行時にフォールバックで強制的に揃えるような処理は入れない（無音で仕様が壊れるのを防ぐため）。

### 2-5. 停止順（push order）

- 停止順は **左→中→右に限定しない**。プレイヤーは任意の順で3つの停止ボタンを押せる（順押し・ハサミ・逆押しすべて可）。
- 各リールの停止判定は、**そのリールの停止ボタンが押された時点の、そのリール自身の位置**のみで決まる（他リールの停止状況に依存しない）。
  - 例外的に「すでに停止した他リールの出目によって、これから止めるリールの引き込み先が変わる」必要があるのは、**複数リールにまたがる役を揃える／避ける**ためである。これを扱うのが §5-3 の「ライン評価に基づく停止位置決定」である。

## 3. データ定義

### 3-1. 図柄（Symbol）

既存の7種を使用する。型は既存の Symbol 型（discriminated union / 文字列リテラル union）を流用すること。存在しない場合は以下で新規定義する。

```ts
export type Symbol =
  | 'SEVEN'   // 7      : BIG図柄
  | 'BAR'     // BAR    : REG図柄（右リールでのREG成立に使用）
  | 'JACK'    // JACK   : 10枚役
  | 'GEM'     // BLUE GEM : 8枚役
  | 'BELL'    // BELL   : 14枚役
  | 'CHERRY'  // CHERRY : 2枚役（角・中段）
  | 'REPLAY'; // REPLAY : 再遊技
```

> 既存コードで `BLUE GEM` を `GEM` 以外（例: `BLUE_GEM`）で表現している場合は既存名に合わせる。**本書中の `GEM` は読み替え可。命名は既存実装を優先。**

### 3-2. リール配列（reel strips）

各リールを「コマ番号 0〜20 の図柄配列」として持つ。

**並び順の定義（最重要）:**
配列インデックス `i` は §2-1 の 0-indexed コマ番号に対応する。配列は **リールが回転して中段を通過していく順**に並べる。すなわち、`position` が 0→1→2→…→20→0 と増えるとき、中段に現れる図柄が `strip[0], strip[1], …, strip[20], strip[0]` の順になるように定義する。

実機リールの「コマ番号」と画面スクロール方向の対応は、使用しているリール画像（リールストリップ画像）のコマ並びと一致させること。**本プロジェクトのリール画像は「上から下へ」コマが並び、回転時は上から下へスクロールして見える。配列の先頭=画像の最上段コマ**とする。

> v7.2 のリール配列表（コマ21が一番上、コマ1が一番下、降順）をそのまま配列化すると順序が反転するので注意。**実装時は、現行コードがすでに保持しているリール配列をそのまま使い、本書では順序を再定義しない**こと。現行コードにリール配列が存在しない場合のみ、リール画像の最上段から順に読み取って配列化し、§3-3 検証を通すこと。

```ts
// 例（構造のみ。中身の図柄は現行コード／リール画像から確定すること）
export const REEL_STRIPS: Record<ReelId, Symbol[]> = {
  left:   [/* 21 symbols, index0 = 画像最上段 */],
  center: [/* 21 symbols */],
  right:  [/* 21 symbols */],
};

export type ReelId = 'left' | 'center' | 'right';
export const REEL_SIZE = 21;
```

### 3-3. 配列設計の制約と検証（必須）

本方式が破綻しないために、リール配列は以下を満たす必要がある。実装と同時に**検証関数を用意し、起動時（開発ビルド時）に assert する**こと。

成立し得る各役について、「盤面上のどの停止位置からでも、すべり0〜4以内にその役を有効ラインに揃えられる位置が存在する」ことを満たすのが理想だが、本機では役ごとに要求レベルを分ける:

- **小役（BELL / JACK / GEM / REPLAY / CHERRY）**: 100%引き込みを保証する。すなわち、各リール単独で見て「任意の押下位置 p（0〜20）について、p からすべり0〜4の範囲（p, p+1, …, p+4 の中段）に、当該小役の構成に使える図柄が最低1つ存在する」ことを検証する。
  - REPLAY は全リールに高頻度で配置されているため通常自動的に満たされる。
  - CHERRY は「角チェリー（上段または下段に CHERRY 停止）」を成立条件とするため、引き込み判定は中段だけでなく上段・下段も対象に含める（§5-2 の角チェリー判定参照）。
- **ボーナス（BIG=SEVEN3つ / REG=SEVEN・SEVEN・BAR）**: 成立時は100%引き込みを保証する。各リールについて「任意の押下位置からすべり0〜4以内に、左=SEVEN、中=SEVEN、右=SEVEN（BIG）／右=BAR（REG）を中段（＝有効ライン上、§3-4）に持ってこられる位置が存在する」ことを検証する。
  - これが満たせない配置の場合、**配列を修正する**こと。実行時フォールバックは設けない（§2-4）。

検証関数の出力イメージ:

```ts
// 開発時のみ実行。失敗したら throw して気づけるようにする。
export function assertReelStripsValid(): void {
  // 各リール・各役について引き込み可能性を全押下位置で総当りチェック
  // 失敗時: `throw new Error('[reel] BIG引き込み不可: rightリール 押下位置12 ...')`
}
```

### 3-4. 有効ライン（payline）

通常時の有効ラインは既存仕様に従う。本書では中段1ライン（左中段・中中段・右中段）を基準ラインとして扱い、上段・下段ライン、斜めラインの有無は**既存の `PAYLINES` 定義を流用**する。

- 既存に `PAYLINES`（評価対象ラインの集合）と `evaluateLines` が存在する前提。停止位置決定（§5-3）はこの `PAYLINES` を参照して「揃う／揃わない」を判定する。
- 既存になければ「中段1ライン」のみで実装を開始してよい（最小構成）。その場合は本書 §8 の受け入れ基準も中段1ラインで判定する。

> 角チェリーだけは「上段 or 下段に CHERRY 停止で成立」という特殊判定（既存 v7.2 §7）。これはラインではなく**左リールのチェリー停止位置**で判定するため、§5-2 で個別に扱う。

## 4. リール状態モデル

### 4-1. リール状態

各リールは以下の状態を持つ。

```ts
export type ReelStatus = 'idle' | 'spinning' | 'stopping' | 'stopped';

export interface ReelState {
  id: ReelId;
  position: number;     // 中段に表示中のコマindex（0..20）。停止中はその位置。回転中は現在の論理位置。
  status: ReelStatus;
  // 回転アニメーション用（実装方式に依存。例: 連続スクロール用のオフセット）
}
```

- 回転中（spinning）は position が時間とともに増え続ける（mod 21）。アニメーションは既存のスクロール方式（translateY / background-position-y）を流用してよい。**論理位置 `position` とアニメーション表示が一致していること**が重要。
- 停止ボタン押下で `stopping` に入り、§5 のアルゴリズムで停止目標 `targetPosition` を計算し、現在位置から `targetPosition` まで滑らせて `stopped` にする。

### 4-2. ゲーム全体の関連状態

本移行で参照／追加する GameState のフィールド（既存 GameState に統合する）:

```ts
interface ReelControlState {
  reels: Record<ReelId, ReelState>;
  // レバーONで確定した成立役（既存の pendingFlag / 成立役フラグを流用）
  // 本書ではこれを stopContext として停止制御から参照する
}
```

成立役の表現は既存の Flag / Role 型（v7.2 のTS移行提案で discriminated union 化予定だったもの）を流用する。停止制御は「今ゲームで何が成立しているか」を読み取れればよい。

## 5. 停止制御アルゴリズム（本書の中核）

### 5-1. 全体フロー

```text
レバーON
 → lottery層が成立役を決定（既存。変更なし）
 → 3リール spinning 開始
 → プレイヤーが任意の順で停止ボタンを押す
     各押下で:
       1. そのリールの「押下時点の position」を取得 = pressPos
       2. そのリールの停止目標 targetPosition を計算（§5-2 / §5-3）
       3. 現在位置→targetPosition まで滑らせて stopped
 → 3リール停止後、出目を確定し evaluateLines で入賞判定（既存の判定流用）
 → 払い出し・状態遷移（既存）
```

### 5-2. 単リールの停止目標計算（基本）

押されたリール `r`、押下位置 `pressPos`、成立役集合 `flags` から停止目標を求める。

候補となる停止位置は `pressPos` から `pressPos + k`（k = 0..MAX_SLIP）の **5通り**。各候補について「そのリールを stop した場合に中段（および角チェリー判定では上下段）に来る図柄」を求め、以下の優先順で目標を選ぶ。

候補位置 `cand(k) = (pressPos + k) % 21`。その中段図柄 `sym(k) = strip[r][cand(k)]`。

**ステップA: 蹴飛ばし対象（絶対に揃えてはいけない図柄）を除外**

- ボーナス非成立時: そのリールで、停止すると**ボーナスが揃ってしまう**位置を候補から除外する。
  - 「揃ってしまう」の判定は単リールだけでは決まらないため、§5-3 のライン評価と統合する。単リール段階では「ボーナス図柄(SEVEN/BAR)を有効ライン上に止めること自体は許可」し、最終的な“3つ揃い”の回避は§5-3で行う。
  - ただし**第3停止**（最後の1リール）では、残り2リールがリーチ状態（例: 左中に SEVEN・SEVEN が中段で揃っている）の場合、ボーナス非成立なら3つ目の SEVEN/BAR を有効ライン上に止めない位置を選ぶ（蹴飛ばし）。
- 小役の取りこぼし回避: ある小役が成立しているとき、その小役を**揃えられる位置を最優先で残す**（ステップB）。

**ステップB: 引き込み対象（揃えるべき図柄）を優先**

成立している役の構成図柄を、できるだけ有効ライン上へ。優先順位（同時成立時の扱いは §5-4）:

1. ボーナス成立時のボーナス図柄（BIG→SEVEN / REG→該当リールの SEVEN or BAR）
2. 中段チェリー成立時の CHERRY（中段に CHERRY を止める。プレミア。§5-5）
3. 一般小役（BELL / JACK / GEM）の構成図柄
4. REPLAY
5. 角チェリー成立時の CHERRY（左リールの上段 or 下段に CHERRY）

各成立役について「k=0..4 の中で、その役の図柄を狙いライン上に置ける最小の k」を求め、最優先の役から順に k を確定する。複数役を同時に満たす k があればそれを選ぶ。両立不能な場合は優先度の高い役を満たす（§5-4）。

**ステップC: 何も該当しなければハズレ目で停止**

- 成立役がない、または当該リールが役に関与しない場合は、**ボーナスもいかなる小役も揃わない安全な位置**を選ぶ。基本は k=0（押した位置）でよいが、k=0 で何かが揃ってしまう（蹴飛ばし必要）場合は、揃わない最小の k を選ぶ。

#### 角チェリーの扱い

角チェリー成立時は、**左リールの上段または下段**に CHERRY を停止させる。候補 k について上段 `(cand(k)-1+21)%21` と下段 `(cand(k)+1)%21` の図柄も見て、どちらかが CHERRY になる最小の k を選ぶ。中段に CHERRY が来てもよいが、本機の角チェリーは上下段停止を成立条件とする（既存 §7）。中段チェリーは別役（§5-5）。

### 5-3. ライン評価に基づく停止位置決定（複数リール整合）

ボーナスの「3つ揃い」を**成立時は揃える／非成立時は揃えない**ために、停止済みリールの出目を考慮する。

- 各リール停止時、**すでに停止しているリールの中段図柄**を読み取り、`PAYLINES`（§3-4）に照らして「この候補位置で止めると、どのラインがどの役で揃うか／揃いかけるか」を計算する。
- ボーナス成立時:
  - まだボーナス図柄を確定できるリールが残っているなら、最終的に有効ライン上で BIG（SEVEN×3）または REG（SEVEN・SEVEN・BAR）が揃う停止位置を選ぶ。
  - 全リールがすべり4以内で揃えられることは §3-3 で保証済み。
- ボーナス非成立時:
  - 第3停止で、残り2リールがボーナスリーチ（有効ライン上に SEVEN・SEVEN など）になっている場合、3つ目のリールは**そのライン上にボーナス完成図柄を置かない**位置を選ぶ（蹴飛ばし）。
  - 蹴飛ばしによって、優先度の低い小役が巻き添えで取りこぼされることは許容する（実機同様、ボーナス回避が最優先で、非成立ボーナスを揃えないことが絶対）。ただし§3-3で小役引き込みも保証しているため、実害が出ないよう配列設計側で吸収する。

実装方針:

```ts
// 疑似コード
function decideStop(reelId, pressPos, flags, stoppedReels): number {
  const cands = [0,1,2,3,4].map(k => ({ k, pos: (pressPos + k) % 21 }));
  // 1. 各候補について、揃う役・揃うボーナスを評価（停止済みリールと合わせて）
  // 2. 蹴飛ばし制約（非成立ボーナスを完成させない）を満たさない候補を除外
  // 3. 引き込み優先度（§5-2 ステップB / §5-4）に従い最良候補を選択
  // 4. 該当なければ安全なハズレ位置（§5-2 ステップC）
  // return 選ばれた pos
}
```

### 5-4. 同時成立・優先順位

複数の役が同時成立し、すべり範囲で全部は揃えられない場合の優先順位（高い順）:

1. ボーナス（BIG / REG）— 成立していれば最優先で揃える
2. 中段チェリー（プレミア。BIG+RUSH確定の特別役。§5-5）
3. 14枚役 BELL
4. 10枚役 JACK
5. 8枚役 GEM
6. REPLAY
7. 角チェリー

> 本機の内部抽選で「ボーナスと小役の同時成立」がどこまで起こるかは lottery 層の仕様による。基本は「ボーナス成立ゲームでも小役を引き込めるなら引き込む」が、停止位置が両立しない場合は上位を優先する。lottery 層が排他制御している役については、ここでの衝突は発生しない。

### 5-5. 中段チェリー（プレミア）

中段チェリー成立時は、左リール中段に CHERRY を停止させる（既存 §7: BIG+RUSH確定のプレミア役）。

- 引き込み優先度は角チェリーより高い（中段に止める）。
- 中段チェリー成立は内部的にボーナス（BIG）+ RUSH 確定を伴う既存仕様に従う。リール表示としては「左中段に CHERRY」を最優先で実現し、ボーナス図柄の引き込みは中段チェリー停止と両立する範囲で行う（既存の演出仕様に従う）。

### 5-6. 目押し（プレイヤー操作）の反映

- 上記アルゴリズムにより、**プレイヤーが押した位置 `pressPos` が起点**になるため、狙ったコマで押せば、すべり0〜4の範囲で狙い図柄が止まりやすくなる（＝目押しが効く）。
- ただし §2-4 の通り「成立役は必ず揃う／非成立役は必ず揃わない」ため、目押しの巧拙で**入賞可否は変わらない**。変わるのは「狙ったラインに止まるか」「すべりコマ数」「演出的な見栄え」のみ。
- ビタ押し（すべり0狙い）成立時はすべり0で止める分岐を必ず持たせる（k=0 が候補に含まれているので自然に満たされる）。

## 7. 演出区間・AUTO との整合

### 7-1. AUTO モード時の停止

AUTO ON 時は人間が押さないため、**システムが擬似的な pressPos を与えて**同じアルゴリズムで停止させる。

- AUTO 時の pressPos は「乱数で 0〜20 を選ぶ」か「各リール一定間隔で進めた現在位置をそのまま使う」かのいずれか。**既定は乱数**（毎ゲーム見た目が変わり自然）。
- AUTO でも引き込み・蹴飛ばしは同じく働くため、成立役は必ず入賞・非成立ボーナスは必ず回避される。
- ボーナス成立ゲームの AUTO: 旧「自動整列」の代わりに、AUTO が擬似 pressPos で停止 → 引き込みで7が揃う。結果は同じ（ボーナス図柄が揃う）だが、経路が統一される。

### 7-2. NEON COUNTDOWN / RUSH継続抽選フェーズ

これらは v7.2 で「BET不要・演出操作のみ・実際の小役抽選やリール停止判定は行わない」区間（§28-9, §28-10）。

- **本移行の対象外**。これらの区間は通常のリール停止アルゴリズム（§5）を**使わない**。演出としてのリール回転/停止アニメは流用してよいが、引き込み・蹴飛ばし・入賞判定は走らせない。
- COUNTDOWN/RUSH の進行ロジック・操作段階は v7.2 のまま維持する。

### 7-3. 入力競合・ウェイト

- 停止ボタンの多重押し・回転開始前の押下などの入力競合制御は、既存の `isProcessing` / 入力ロック仕様（v7.2 §28 系）を流用する。
- ウェイトは v7.2 §28-11 を踏襲（即時回転開始・即時停止。長い強制ウェイトは設けない）。すべりアニメーションは短時間（数十〜100ms程度）で見栄えする範囲に収める。

## 8. 受け入れ基準（テスト）

以下をユニットテスト／結合テストとして用意し、すべて通ること。テストフレームワークは既存に合わせる（Vitest 等）。

### 8-1. 配列検証

- `assertReelStripsValid()` が成功する（全リール・全役・全押下位置で引き込み／蹴飛ばし可能）。
- 各リール配列長が 21。

### 8-2. 引き込み（100%）

- ボーナス（BIG）成立時、**全ての押下位置パターン（左右中それぞれ 0..20、計 21^3 通り、または十分なサンプル）**で、停止後に有効ライン上へ SEVEN×3 が揃う。
- REG 成立時、同様に SEVEN・SEVEN・BAR が揃う。
- 各小役（BELL/JACK/GEM/REPLAY）成立時、当該小役が有効ライン上に揃う（取りこぼし0）。
- 角チェリー成立時、左リール上段または下段に CHERRY が停止する。
- 中段チェリー成立時、左リール中段に CHERRY が停止する。

### 8-3. 蹴飛ばし（100%）

- ボーナス非成立時、**いかなる押下位置・押下順**でも、有効ライン上にボーナス（SEVEN×3 / SEVEN・SEVEN・BAR）が揃わない。
- 成立していない小役が、有効ライン上に揃わない（誤入賞0）。

### 8-4. すべり範囲

- すべてのケースで、各リールのすべりコマ数が 0〜MAX_SLIP(=4) の範囲に収まる（5コマ以上滑らない）。
- すべりは常に進行方向（position 増加方向）のみ。

### 8-5. 目押し反映

- 同じ成立役でも、異なる pressPos を与えると停止出目（position）が変わり得ることを確認（押し位置が結果に反映される）。
- ビタ押し（k=0 で揃う位置を押下）時、すべり0で停止する。

### 8-6. 停止順非依存

- 順押し・逆押し・ハサミ押しのいずれでも、§8-2/§8-3 の引き込み・蹴飛ばしが成立する。

### 8-7. 既存回帰

- lottery 層の抽選結果・確率・設定差が本変更で変わっていない。
- 配当・天井・RUSH/COUNTDOWN 進行が本変更前と同一挙動。
- AUTO モードで成立役が必ず入賞し、非成立ボーナスが必ず回避される。
